# PHP 文档编写内容与风格规范总结

## 1. 核心编写原则 (Core Principles)

- **客观性 (Objectivity)**：文档必须保持客观、严谨、简洁。严禁使用任何情感化修饰语（如：伟大的、惊人的、极其、极致）。
- **去隐喻化 (No Metaphors)**：禁止使用非技术性的比喻（如：灵魂、心脏、英雄、基石、天花板）。
- **去黑话化 (No Jargon)**：禁止使用营销类或非技术背景的词汇（如：赋能、闭环、抓手、工业级实战、秘密武器、阿里味词汇）。
- **零起点逻辑 (Zero-Start Logic)**：文档应适合从零开始学习。所有技术概念必须提供详尽的逻辑推导和原理分析，确保读者能建立完整的知识框架。
- **专业技术标准 (Professional Standards)**：文档必须符合专业技术文档标准，内容详实且完整。

## 2. 内容完整性要求 (Content Completeness - 12 要素)

每个章节文档必须完整覆盖以下 12 个要素：

| 序号 | 要素名称         | 逻辑说明                                         |
|:-----|:-----------------|:-------------------------------------------------|
| 1    | **概述**         | 准确定义技术点，说明其在系统中的定位。           |
| 2    | **特性**         | 列表说明核心技术优势与功能特点。                 |
| 3    | **语法 / 架构**  | 提供物理层面的架构图或核心语法声明。             |
| 4    | **参数详述**     | 穷尽式解析 API 参数、配置项及环境变量（表格化）。|
| 5    | **返回值 / 状态** | 明确标注函数返回值类型及系统反馈的状态码信息。   |
| 6    | **代码示例**     | 包含完整、可运行的代码片段（带文件路径标注）。   |
| 7    | **输出结果说明** | 记录代码运行后的真实控制台输出及其逻辑解析。     |
| 8    | **注意事项**     | 描述边缘情况、异常处理逻辑及跨平台差异。         |
| 9    | **常见问题 (FAQ)**| 针对学习中的疑惑提供针对性的逻辑解答。           |
| 10   | **最佳实践**     | 提供工业级的避坑指南和架构建议。                 |
| 11   | **对比分析**     | 与相似技术或其他实现方式的性能对比。             |
| 12   | **练习任务**     | 设计 3-5 个具有逻辑深度的调研或实践任务。        |

## 3. Markdown 格式规范 (Markdown Standards)

- **标题空行**：所有的 `#` 标题前后**必须且仅有**一行空行。
- **段落与列表**：段落、代码块、列表前后必须各留一行空行。严禁连续出现两行空行。
- **标准列表**：列表符号（`*` 或 `-`）后仅保留一个空格。
- **表格物理对齐**：Markdown 源代码中的 `|` 符号必须垂直对齐。
  - **对齐算法**：参考"9 个中文字符宽度 ≈ 15 个西文字符宽度"比例进行空格补偿。
  - **效果要求**：在文本编辑器（如 VS Code 或 Vim）的纯文本模式下必须视觉整齐。
- **代码块标注**：必须注明语言（如 `php`, `json`, `bash`, `sql`），并在顶部注入 `// 文件:` 和 `// 功能:` 元数据。
- **去 Emoji 化**：除特殊技术符号外，严禁在文档中使用任何装饰性 Emoji。

## 4. 技术栈规范 (Technical Stack)

- **PHP 版本**：所有代码示例必须使用 PHP 8.2+ 语法特性。
- **类型声明**：强制使用类型声明（参数类型、返回类型、属性类型）。
- **PSR 标准**：代码必须符合 PSR-12 编码规范。
- **命名空间**：使用 PSR-4 自动加载标准。
- **现代特性**：优先使用 PHP 8+ 新特性（如：构造器属性提升、枚举、只读属性等）。

## 5. 关联与引用 (References)

- **避免冗余**：当涉及 Web 前端基础知识时，应通过链接进行关联，不进行重复解释。
- **交叉引用**：合理使用章节间的交叉引用，帮助读者建立知识体系。
