# 文档组织规范

## 概述

本文档定义了文档的组织结构、命名规范、创建流程等。**文档创建流程是从零开始或对旧版内容进行重构的关键流程，必须严格遵守。**

---

## 目录结构

- 按阶段、章、节拆分独立目录
- 格式：`docs/stage-xx/chapter-yy/readme.md`
- 根目录 `readme.md` 保留完整阶段与章节目录，并指向对应文档

---

## 文件命名规范

**所有文件**（包括 readme.md）必须使用：

- 小写字母、数字和连字符（-）
- **目录名**同样遵循小写字母、数字和连字符（-）的规范
- **扩展名**统一为 `.md`
- **阶段、章节和小节文档**统一命名为 `readme.md`（小写），不使用 `README.md`

**示例**：

- ✅ 正确：`stage-01-foundation`、`chapter-01-history`、`section-01-history.md`、`readme.md`
- ❌ 错误：`Stage-01-Foundation`、`README.md`、`section_01_history.md`

## 文档创建流程

### 步骤 1：创建阶段和章节大纲

**目标**：在开始创建具体文档之前，先确定整个项目的阶段划分、章节结构和大纲。

**要求**：

#### 1.1 面向读者和基础要求

- 明确目标读者群体（前端开发者转后端、全栈开发初学者、其他语言开发者等）
- 确定前置知识要求（必须掌握、建议掌握、不需要掌握）
- 明确学习环境要求（操作系统、硬件、软件）
- 确保内容面向零基础学员，但假设读者具备基础编程思维

#### 1.2 学习目标和学习路径

- 明确整体学习目标（完成本指南后应具备的能力）
- 设计合理的学习路径（从基础到高级，循序渐进）
- 确保阶段之间的依赖关系清晰
- 确保每个阶段都有明确的学习目标和完成标准
- 设计检查点，帮助学员确认是否具备继续学习的条件

#### 1.3 对内容的要求

**语言版本**：

- 使用最新稳定版本的语法特性（优先使用最新特性）
- 必须使用类型声明（如适用）
- 必须符合官方编码规范
- 必须包含必要的注释（如适用）
- 优先使用现代特性

**工具版本**：

- 使用最新稳定版本
- 明确版本要求，避免版本兼容性问题

**流行度**：

- 优先介绍主流、广泛使用的技术和工具
- 避免介绍过时或不再维护的项目

**框架和库**：

- 优先介绍主流框架和库
- 优先介绍广泛使用的库和工具
- 避免介绍过时或不再维护的项目

**相关技术领域**：

- 明确相关技术领域的覆盖范围
- 确保技术选型合理

**应用场景**：

- 提供实际应用场景和案例
- 说明技术在实际项目中的应用
- 提供完整的项目实践示例

**文档语言**：

- 使用简体中文（或根据目标受众选择）
- 客观、严谨、简洁
- 禁止使用情感化修饰语
- 禁止使用非技术性比喻
- 禁止使用营销类词汇

#### 1.4 结构规划要求

- 确定所有阶段的划分和定位
- 确定每个阶段包含的章节
- 确定每个章节包含的小节
- 确保阶段之间的依赖关系合理
- 确保学习路径循序渐进
- 列出阶段和章节的编号、名称、简要说明

**输出**：

- 阶段大纲文档（包含所有阶段的列表和说明）
- 章节大纲文档（包含所有章节的列表和说明）
- 项目结构规划文档
- 学习路径规划文档

**注意事项**：

- 此步骤是后续所有工作的基础
- 需要充分考虑学习路径的合理性
- 需要确保内容归类合理，避免重复
- 需要确保技术选型符合项目要求（版本、流行度、维护状态等）
- 大纲确定后，后续步骤应严格按照大纲执行

---

### 步骤 2：创建完整目录结构

**要求**：

- 创建所有阶段、章节、小节的目录结构
- 目录命名：小写字母、数字、连字符（-）
- 示例：`docs/stage-01-foundation/chapter-01-history/section-01-history.md`
- **禁止使用脚本批量操作**，必须逐个创建

---

### 步骤 3：完成阶段总览文档

**文件位置**：`docs/stage-XX-xxx/readme.md`

**必须包含内容**：

- 阶段标题和描述
- 定位（本阶段的定位和作用）
- 前置知识要求（必须掌握、不需要掌握、如何检查）
- 学习时间估算
- 练习建议（基础练习、综合练习）
- 章节内容列表（包含链接和简要说明）
- 完成本阶段后的能力
- 相关章节（与其他阶段的关联）

---

### 步骤 4：完成章节总览文档

**文件位置**：`docs/stage-XX-xxx/chapter-XX-xxx/readme.md`

**必须包含内容**：

- 章节标题
- 目标（3-5 条学习目标）
- 章节内容（说明分为几个小节，列出每个小节的标题、链接和简要说明）
- 核心概念/特性/概览（根据章节类型选择）
- 学习建议（按顺序学习、重点掌握、实践练习）
- 完成本章后的能力
- 相关章节（与其他章节的关联）

---

### 步骤 5：完成小节结构标题和大纲

**文件位置**：`docs/stage-XX-xxx/chapter-XX-xxx/section-XX-xxx.md`

**重要**：**不填充完整内容**，只创建结构标题和详细大纲

**必须包含**：

- 概述（含章节类型，如果适用）
- 详细内容大纲（列出主要章节标题和内容说明）
- 必需要素说明（根据章节类型，参考 [内容规范](04-content-standards.md)）
- 基本用法示例（简要）
- 使用场景
- 注意事项
- 常见问题
- 最佳实践

**章节类型**：根据 [内容规范](04-content-standards.md) 确定章节类型

---

### 步骤 6：完成小节的详细内容

**要求**：

#### 6.1 新引入概念的详细解释

- 对于新引入的概念，应该详细解释其定义、作用、原理
- 提供充分的背景知识和上下文
- 确保零基础学员能够完全理解

#### 6.2 第三方工具和库的简单介绍

- 对于第三方工具和库，只需要简单介绍其用途和基本用法
- 不需要深入讲解其内部实现
- 可以引用官方文档和线上资源

#### 6.3 内置模块/类/方法的细致介绍

- 对于内置模块、类、方法，应该细致介绍
- **必须包含**：语法、参数、返回值（三个部分缺一不可）
- 添加必要的示例代码，确保代码完整可运行
- 提供使用场景和注意事项

#### 6.4 避免内容重复

- 注意内容不要重复，重复的内容应该引用其他已有的章节
- 使用交叉引用链接指向相关章节
- 避免在同一文档中重复解释相同概念

#### 6.5 允许引用外部资源

- 允许引用线上资源、文档、文章等内容
- 提供准确的链接和引用说明
- 对于外部资源，简要说明其内容和相关性

#### 6.6 核对编号和路径

- 注意核对编号、路径，确保编号从 1 开始，遵循 1.2.3 结构
- 及时更新相关的目录文档（如 `readme.md`）
- 确保所有链接和引用路径正确

#### 6.7 Markdown 格式规范

- **严格要求**：所有输出的文档内容必须严格遵守标准的 Markdown 规范
- 参考 [格式规范](03-format-standards.md)

#### 6.8 内容完整性

- 必须符合 [内容规范](04-content-standards.md)
- 根据章节类型确定必需要素（参考 [内容规范](04-content-standards.md)）
- 内容详实完整，参考 300-500 行（基础概念 200-300 行，重要概念 300-500 行，复杂概念 500-800 行）

#### 6.9 代码规范

- 所有代码必须符合 [格式规范](03-format-standards.md)
- 必须符合 PSR-1 和 PSR-12 编码规范
- 使用最新稳定版本语法
- 必须包含 `declare(strict_types=1);`
- 纯 PHP 文件不得使用结束标签 `?>`

## 小节文档结构模板

```markdown
# X.X.X 小节标题

## 概述

（简要说明本节内容，面向零基础学员）

**章节类型**：概念性章节 / 配置性章节 / 语法性章节 / 工具性章节 / 实践性章节 / 参考性章节 / 最佳实践章节 / 对比性章节

**主要内容**：
- 内容点 1
- 内容点 2
- 内容点 3

## 特性

（列出关键特性、优势、特点）

## 语法/定义

（根据章节类型选择合适的内容：语法、架构、定义等）

## 参数详述

（如果是函数/方法/API，必须包含参数详述）

## 返回值/状态

（如果是函数/方法/API，必须包含返回值说明）

## 基本用法

（简要示例，确保代码完整可运行）

## 完整示例

（完整的、可运行的代码示例）

## 输出结果说明

（记录代码运行后的真实输出及其逻辑解析）

## 使用场景

（提供实际应用场景和案例）

## 注意事项

（描述边缘情况、异常处理逻辑及跨平台差异）

## 常见问题 (FAQ)

（针对学习中的疑惑提供针对性的逻辑解答）

## 最佳实践

（提供工业级的避坑指南和架构建议）

## 对比分析

（与相似技术或其他实现方式的性能对比）

## 练习任务

（设计 3-5 个具有逻辑深度的调研或实践任务）
```

**注意**：不是所有章节都需要全部 12 个要素，根据章节类型确定必需要素（参考 [内容规范](04-content-standards.md)）。

## 阶段文档要求

每个阶段的 `readme.md` 必须包含：

- 前置知识要求
- 学习时间估算
- 练习建议
- 章节内容列表

## 文件完整性检查

- 创建新章节前，必须检查根目录 `readme.md` 中的目录结构
- 确保所有列出的章节文件都已创建
- 如发现缺失文件，必须补全

## 质量保证

### 禁止事项

1. **禁止批量操作**：所有文件必须逐个创建，不使用脚本
2. **禁止占位符**：不要在文档中使用"（待填充）"等占位符
3. **必须提供大纲**：即使不填充内容，也必须提供详细大纲
4. **禁止编号从 0 开始**：所有编号从 1 开始，不使用 0

### 内容质量要求

1. **面向零基础**：所有内容必须面向零基础学员
2. **详实完整**：内容必须详实完整，参考 300-500 行
3. **12 要素**：根据章节类型确定必需要素（不是所有章节都需要全部 12 个要素）
4. **代码规范**：所有代码必须符合官方编码规范，使用最新稳定版本语法

### 格式规范要求

**⚠️ Markdown 格式规范**：所有输出的文档内容必须严格遵守标准的 Markdown 规范。

1. **文件命名**：统一使用小写字母、数字、连字符（-），统一使用小写 `readme.md`
2. **表格格式**：表格必须在源文件中对齐，使用对齐标记（`|:---|`），宽度比例：9 中文 ≈ 15 英文
3. **代码块格式**：使用标准 markdown 代码块（带语言标签），禁止使用 CODE REFERENCES 格式
4. **标题空行**：标题前后必须空行
5. **代码块空行**：代码块前后必须空行
6. **段落空行**：段落之间必须空行
