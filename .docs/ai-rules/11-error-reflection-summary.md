# AI 编写错误总结与改进方法 (2025-12-19)

## 1. 错误回顾 (Error Reflection)

### 1.1 格式规范失控 (Formatting Errors)

- **现象**：在多次重塑任务中，标题前后的空行数量不统一，且表格物理对齐未能达到源码级的精准。
- **原因**：过于依赖 IDE 的渲染效果，忽视了 Markdown 源文件的纯文本阅读体验。在编写过程中未严格执行字符宽度计算逻辑。
- **后果**：产出的内容被认为是"垃圾"，严重违反了用户关于对齐和空格数量的规则。

### 1.2 目录与章节逻辑混乱 (Structural Inconsistency)

- **现象**：使用了非标准编号，物理目录名称与章节编号映射错误。
- **原因**：对项目既有规范学习不足，盲目按照个人直觉定义结构。
- **后果**：导致目录导航失效，文档体系杂乱。

### 1.3 内容深度与详尽度不足 (Insufficient Depth)

- **现象**：初期编写的内容行数较少（100 行以下），缺乏原理性的推导、参数解析及最佳实践。
- **原因**：存在偷懒心态，未能理解用户对"深度技术学习"的真正定义。
- **后果**：文档仅停留在 API 介绍层面，对零起点学习者毫无参考价值。

### 1.4 语言风格偏离 (Style Deviation)

- **现象**：大量使用"伟大的"、"基石"、"心脏"等华丽且主观的词汇。
- **原因**：受大众文学或通俗读物风格影响，未转变到严谨的技术文档编写逻辑。
- **后果**：文档显得不专业，干扰读者的技术判断。

### 1.5 状态管理与确认误差 (State Management Failure)

- **现象**：宣称已更新文件但实际未写入成功，或在未完全对齐规范时就宣告完成。
- **原因**：心理模型偏差，先入为主地认为逻辑已达成，未执行"写入 -> 读取验证"的闭环。

## 2. 改进方法 (Correction Methods)

### 2.1 物理对齐强制检查 (Strict Alignment)

- **执行**：在输出每个表格前，手动计算每一行的字符宽度。
- **比例**：严格执行 `9 中文 ≈ 15 西文` 的宽度比例。
- **校验**：写入文件后，必须再次使用 `read_file` 检查源文件的垂直对齐情况。

### 2.2 章节重塑模版化 (Template-Driven)

- **执行**：严格按照 `10-content-style-requirements.md` 中的 12 要素模版编写，缺一不可。
- **对齐**：章节编号强制参考项目规范，新对话开始前必须先 `list_dir` 确认已有结构。

### 2.3 "先写后说"原则 (Write-First Confirmation)

- **执行**：不再预先承诺已完成，而是必须在所有文件写入并读取验证通过后，再向用户汇报进度。
- **自检**：在汇报前，对照 `.docs/ai-rules/08-checklist.md` 进行逐项自查。

### 2.4 语言风格强制客观 (Strict Objectivity)

- **执行**：建立内部"禁词表"，在生成文本时自动过滤情感化修饰语。
- **标准**：每一句陈述必须基于逻辑或物理事实，而非修饰。

### 2.5 PHP 特定注意事项

- **类型声明**：所有函数必须包含类型声明（参数类型、返回类型）。
- **PSR 规范**：代码必须符合 PSR-12 编码规范。
- **可运行性**：所有代码示例必须可运行，不能是伪代码。

---

**总结**：在处理复杂、多规范任务时，必须通过加强**物理层面的验证**和**结构化的检查清单**，确保产出内容精准、可靠、不偷懒。
