## 一、详细规则总结

### 1.1 核心执行规则

#### 基础规则
- **对话语言**：始终使用简体中文
- **模型声明**：每次回复开头必须标注当前模型/服务
- **记忆要求**：当用户提到"记住、保存、记录、牢记"等词语时，应当更新 memory 并记录到合适的文档中

#### 执行工作流
- **执行前**：先阅读规则和文档
- **执行时**：严格遵守相关规则
- **执行后**：进行分析评估，避免违反规则

### 1.2 格式规范

#### Markdown 格式规范

**⚠️ 严格要求**：所有输出的文档内容必须严格遵守标准的 Markdown 规范。详见报告开头的"重要提醒"部分和 [Markdown 格式规范](.docs/ai-rules/03-markdown-format.md)。

#### 代码格式规范

#### 代码格式规范
- **PHP 版本**：所有代码示例必须使用 PHP 8.2+ 语法特性
- **类型声明**：强制使用类型声明（参数类型、返回类型、属性类型）
- **PSR 标准**：代码必须符合 PSR-12 编码规范
- **代码注释**：提供 PHPDoc 注释（如适用）
- **可运行性**：所有代码示例必须完整可运行

### 1.3 内容规范

#### 核心编写原则
- **客观性**：文档必须保持客观、严谨、简洁，严禁使用情感化修饰语
- **去隐喻化**：禁止使用非技术性的比喻（如：灵魂、心脏、英雄、基石）
- **去黑话化**：禁止使用营销类或非技术背景的词汇（如：赋能、闭环、抓手、阿里味词汇）
- **零起点逻辑**：文档应适合从零开始学习，所有技术概念必须提供详尽的逻辑推导和原理分析

#### 受众定位
- **所有文档面向零基础学员撰写**
- 语言需循序渐进但内容必须详实完整
- 避免过度简化，确保零基础学员能够完全理解
- 每个概念都要有充分的解释和多个示例

### 1.4 文档组织规范

#### 文件命名
- **所有文件**：小写字母、数字、连字符（-）
- **统一使用**：`readme.md`（小写），不使用 `README.md`
- **目录名**：同样遵循小写字母、数字、连字符（-）的规范

#### 目录结构
- 按阶段、章、节拆分独立目录
- 格式：`docs/stage-xx/chapter-yy/readme.md`
- 根目录 `readme.md` 保留完整阶段与章节目录，并指向对应文档

### 1.5 禁止事项

- ❌ 禁止使用脚本批量操作
- ❌ 禁止使用 CODE REFERENCES 格式
- ❌ 禁止编号从 0 开始
- ❌ 禁止在概述中使用"（待填充）"等占位符
- ❌ 禁止在小节中只写"（待填充）"而不提供大纲
- ❌ 禁止通过 `run_terminal_cmd` 直接修改文件，应该展示改动范围供审阅

## 二、内容结构和详细程度要求

### 2.1 编号规则

#### 基本规则
- **所有编号从 1 开始，不使用 0**
- **遵循 1.2.3 这种结构**（阶段.章节.小节）
- **示例**：
  - ✅ 正确：1.1、1.2、1.3
  - ❌ 错误：1.0、1.0.1、1.0.2

#### 编号层级
- **阶段编号**：1、2、3...（阶段一、阶段二、阶段三）
- **章节编号**：1.1、1.2、1.3...（阶段一的第1、2、3章）
- **小节编号**：1.1.1、1.1.2、1.1.3...（第1章的第1、2、3节）

### 2.2 内容详细程度

#### 目标要求
- **目标**：充分详实、零基础可学
- **行数参考**：300-500 行作为详实度范例，视主题复杂度可增减
- **所有概念**：都要有充分的解释
- **所有语法**：都要有完整说明
- **所有代码示例**：都要完整可运行

### 2.3 内容完整性要求

#### 12 要素参考列表

以下 12 个要素是内容完整性的参考标准，但**并非所有章节都必须包含全部 12 个要素**。根据章节类型的不同，有些要素是必需的，有些是可选的，有些甚至是不需要的：

| 序号 | 要素名称         | 逻辑说明                                         |
|:-----|:----------------|:-------------------------------------------------|
| 1    | **概述**         | 准确定义技术点，说明其在系统中的定位。           |
| 2    | **特性**         | 列表说明核心技术优势与功能特点。                 |
| 3    | **语法 / 架构**  | 提供物理层面的架构图或核心语法声明。             |
| 4    | **参数详述**     | 穷尽式解析 API 参数、配置项及环境变量（表格化）。|
| 5    | **返回值 / 状态** | 明确标注函数返回值类型及系统反馈的状态码信息。   |
| 6    | **代码示例**     | 包含完整、可运行的代码片段（带文件路径标注）。   |
| 7    | **输出结果说明** | 记录代码运行后的真实控制台输出及其逻辑解析。     |
| 8    | **注意事项**     | 描述边缘情况、异常处理逻辑及跨平台差异。         |
| 9    | **常见问题 (FAQ)**| 针对学习中的疑惑提供针对性的逻辑解答。           |
| 10   | **最佳实践**     | 提供工业级的避坑指南和架构建议。                 |
| 11   | **对比分析**     | 与相似技术或其他实现方式的性能对比。             |
| 12   | **练习任务**     | 设计 3-5 个具有逻辑深度的调研或实践任务。        |

#### 章节类型决定必需要素

**重要**：必须根据章节类型（概念性、配置性、语法性、工具性、实践性、参考性、最佳实践、对比性）来确定哪些要素是必需的，哪些是可选的。详见 [内容结构标准](.docs/ai-rules/06-content-structure.md)。

### 2.4 方法/函数/API 说明规范

**介绍任何内置方法、函数、模块时，必须包含**：

#### 必需格式（三个部分，缺一不可）

1. **语法**（必需）：
   ```markdown
   **语法**：`functionName(type $param1, type $param2 = default): returnType`
   ```

2. **参数**（必需）：
   ```markdown
   **参数**：
   - `$param1`：参数说明
   - `$param2`：可选，参数说明
   ```

3. **返回值**（必需）：
   ```markdown
   **返回值**：返回值的详细说明。
   ```

### 2.5 章节类型与结构要求

根据章节类型，确定必需要素和可选要素：

#### 概念性章节（Conceptual）
- **必需要素**：概述、核心内容、总结
- **可选要素**：特性、常见问题、相关资源

#### 配置性章节（Configuration）
- **必需要素**：概述、特性、配置步骤/语法、注意事项、常见问题
- **可选要素**：输出结果说明、对比分析、最佳实践

#### 语法性章节（Syntax）
- **必需要素**：概述、特性、语法/定义、基本用法、代码示例、使用场景、注意事项、最佳实践
- **可选要素**：参数详情、返回值/状态、输出结果说明、对比分析、常见问题

#### 工具性章节（Tool）
- **必需要素**：概述、特性、基本用法/命令、使用场景、注意事项、常见问题、最佳实践
- **可选要素**：输出结果说明、对比分析、参数详情

#### 实践性章节（Practical）
- **必需要素**：概述、实践步骤、完整代码示例、输出结果说明、注意事项、练习任务
- **可选要素**：特性、常见问题、最佳实践

#### 参考性章节（Reference）
- **必需要素**：概述、定义/语法、示例、使用场景
- **可选要素**：特性、参数详情、返回值/状态、对比分析、注意事项

#### 最佳实践章节（Best Practices）
- **必需要素**：概述、实践建议/原则、代码示例、最佳实践、注意事项
- **可选要素**：特性、对比分析、常见问题、输出结果说明

#### 对比性章节（Comparative）
- **必需要素**：概述、对比分析、使用建议、注意事项
- **可选要素**：特性、代码示例、常见问题、最佳实践

## 三、阶段和章节内容建立的操作流程

### 3.1 文档创建顺序

#### 步骤 1：创建阶段和章节大纲

**目标**：在开始创建具体文档之前，先确定整个项目的阶段划分、章节结构和大纲。

**要求**：

1. **面向读者和基础要求**
   - 明确目标读者群体（前端开发者转后端、全栈开发初学者、其他语言开发者等）
   - 确定前置知识要求（必须掌握、建议掌握、不需要掌握）
   - 明确学习环境要求（操作系统、硬件、软件）
   - 确保内容面向零基础学员，但假设读者具备基础编程思维

2. **学习目标和学习路径**
   - 明确整体学习目标（完成本指南后应具备的能力）
   - 设计合理的学习路径（从基础到高级，循序渐进）
   - 确保阶段之间的依赖关系清晰
   - 确保每个阶段都有明确的学习目标和完成标准
   - 设计检查点，帮助学员确认是否具备继续学习的条件

3. **对内容的要求**

   - **PHP 语言版本**：
     - PHP 8.2+ 语法特性（优先使用最新特性）
     - 必须使用类型声明
     - 必须符合 PSR-12 编码规范
     - 必须包含 PHPDoc 注释（如适用）
     - 优先使用现代 PHP 特性（构造器属性提升、枚举、只读属性等）

   - **工具版本**：
     - MySQL 最新稳定版本
     - Composer 最新版本
     - 其他工具使用最新稳定版本

   - **流行度**：
     - 优先介绍主流、广泛使用的技术和工具
     - 避免介绍过时或不再维护的项目

   - **框架和库**：
     - 优先介绍主流框架（Laravel、Symfony 等）
     - 优先介绍广泛使用的库和工具
     - 避免介绍过时或不再维护的项目

   - **相关技术领域**：
     - Web 开发（HTTP、RESTful API、会话管理、认证授权等）
     - 数据库操作（PDO、ORM、缓存等）
     - 系统编程（文件操作、网络编程、CLI 开发等）
     - 架构设计（SOLID 原则、设计模式、MVC、DDD 等）
     - DevOps（Docker、CI/CD、部署等）

   - **应用场景**：
     - 提供实际应用场景和案例
     - 说明技术在实际项目中的应用
     - 提供完整的项目实践示例

   - **文档语言**：
     - 使用简体中文
     - 客观、严谨、简洁
     - 禁止使用情感化修饰语
     - 禁止使用非技术性比喻
     - 禁止使用营销类词汇

4. **结构规划要求**
   - 确定所有阶段的划分和定位
   - 确定每个阶段包含的章节
   - 确定每个章节包含的小节
   - 确保阶段之间的依赖关系合理
   - 确保学习路径循序渐进
   - 列出阶段和章节的编号、名称、简要说明

**输出**：
- 阶段大纲文档（包含所有阶段的列表和说明）
- 章节大纲文档（包含所有章节的列表和说明）
- 项目结构规划文档
- 学习路径规划文档

**注意事项**：
- 此步骤是后续所有工作的基础
- 需要充分考虑学习路径的合理性
- 需要确保内容归类合理，避免重复
- 需要确保技术选型符合项目要求（版本、流行度、维护状态等）
- 大纲确定后，后续步骤应严格按照大纲执行

#### 步骤 2：创建完整目录结构
- 创建所有阶段、章节、小节的目录结构
- 目录命名：小写字母、数字、连字符（-）
- 示例：`docs/stage-01-foundation/chapter-01-history/section-01-history.md`
- **禁止使用脚本批量操作**，必须逐个创建

#### 步骤 3：完成阶段总览文档
- 文件：`docs/stage-XX-xxx/readme.md`
- 包含内容：
  - 阶段标题和描述
  - 定位
  - 前置知识要求（必须掌握、不需要掌握、如何检查）
  - 学习时间估算
  - 练习建议（基础练习、综合练习）
  - 章节内容列表（包含链接和简要说明）
  - 完成本阶段后的能力
  - 相关章节

#### 步骤 4：完成章节总览文档
- 文件：`docs/stage-XX-xxx/chapter-XX-xxx/readme.md`
- 包含内容：
  - 章节标题
  - 目标（3-5 条学习目标）
  - 章节内容（说明分为几个小节，列出每个小节的标题、链接和简要说明）
  - 核心概念/特性/概览（根据章节类型选择）
  - 学习建议（按顺序学习、重点掌握、实践练习）
  - 完成本章后的能力
  - 相关章节

#### 步骤 5：完成小节结构标题和大纲
- 文件：`docs/stage-XX-xxx/chapter-XX-xxx/section-XX-xxx.md`
- **不填充完整内容**，只创建结构标题和详细大纲
- 包含：
  - 概述（含章节类型）
  - 详细内容大纲（列出主要章节标题和内容说明）
  - 必需要素说明（根据章节类型）
  - 基本用法示例（简要）
  - 使用场景
  - 注意事项
  - 常见问题
  - 最佳实践

#### 步骤 6：完成小节的详细内容（暂不执行）

**状态**：当前阶段暂不执行，等待步骤 1-4 全部完成并审核通过后再进行。

**要求**：

1. **新引入概念的详细解释**
   - 对于新引入的概念，应该详细解释其定义、作用、原理
   - 提供充分的背景知识和上下文
   - 确保零基础学员能够完全理解

2. **第三方工具和库的简单介绍**
   - 对于第三方工具和库，只需要简单介绍其用途和基本用法
   - 不需要深入讲解其内部实现
   - 可以引用官方文档和线上资源

3. **内置模块/类/方法的细致介绍**
   - 对于 PHP 内置模块、类、方法，应该细致介绍
   - 必须包含：语法、参数、返回值（三个部分缺一不可）
   - 添加必要的示例代码，确保代码完整可运行
   - 提供使用场景和注意事项

4. **避免内容重复**
   - 注意内容不要重复，重复的内容应该引用其他已有的章节
   - 使用交叉引用链接指向相关章节
   - 避免在同一文档中重复解释相同概念

5. **允许引用外部资源**
   - 允许引用线上资源、文档、文章等内容
   - 提供准确的链接和引用说明
   - 对于外部资源，简要说明其内容和相关性

6. **核对编号和路径**
   - 注意核对编号、路径，确保编号从 1 开始，遵循 1.2.3 结构
   - 及时更新相关的目录文档（如 `readme.md`）
   - 确保所有链接和引用路径正确

7. **Markdown 格式规范**
   - **严格要求**：所有输出的文档内容必须严格遵守标准的 Markdown 规范
   - 详见报告开头的"重要提醒"部分和 [Markdown 格式规范](.docs/ai-rules/03-markdown-format.md)
